# Fraud Detection Pipeline

End-to-end fraud detection system built on the [Kaggle Credit Card Fraud dataset](https://www.kaggle.com/datasets/mlg-ulb/creditcardfraud) (284,807 transactions, 0.17% fraud rate). Covers data-quality validation, threshold tuning, KPI tracking, and production-style SQL monitoring queries.

---

## What This Project Does

Most fraud detection tutorials stop at model accuracy. This project goes further — it replicates the operational layer around the model: threshold tuning that makes the precision/recall trade-off explicit, KPI tracking, and SQL monitoring queries that flag threshold drift and SLA breaches. All reported numbers are produced by the committed code on the real dataset.

## Key Techniques

| Area | Approach |
|---|---|
| Data Quality | Row-level checks — duplicates removed, negative amounts, class validity |
| Leakage Control | Amount/Time scaled **after** the split (scaler fit on train only) |
| Class Imbalance | `class_weight='balanced'`; stratified train/test split |
| Threshold Tuning | Grid search 0.10–0.89 to maximise F1; trade-off vs default 0.50 quantified |
| Models | Logistic Regression (interpretable baseline) + Random Forest (production) |
| KPI Tracking | Recall, precision, false-positive rate, F1, ROC-AUC |
| SQL Monitoring | Threshold drift (4-week rolling avg), SLA compliance, chargeback anomalies |

## Results

Random Forest, class-weighted, `random_state=42`, on the deduped dataset (283,726 rows; 95 fraud cases in the held-out test set).

| Metric | Default Threshold (0.50) | Tuned Threshold (0.22) |
|---|---|---|
| F1 Score | 0.82 | 0.85 |
| Recall (fraud caught) | 71% | 78% |
| Precision | 97% | 94% |
| False Positives (test set) | 2 | 5 |
| ROC-AUC | 0.92 | 0.92 |

**Reading the trade-off:** lowering the decision threshold from 0.50 to 0.22 raises recall (71% → 78%) and F1 (0.82 → 0.85). It also *increases* false positives (2 → 5), because catching more fraud means flagging more borderline cases — the standard precision/recall trade. In fraud, a missed case usually costs more than a reviewed one, which is why the lower threshold is preferred here.

> Numbers are deterministic for a given scikit-learn version. Re-running reproduces the committed charts and `kpi_summary.csv`.

## SQL Queries

`sql/fraud_kpi_queries.sql` contains five monitoring queries, written in **Snowflake** SQL:

1. **Daily Fraud KPI Summary** — fraud rate, recovery rate, case turnaround by day
2. **Threshold Drift Detection** — flags accounts drifting >1.5% from their 4-week rolling average
3. **False Positive Analysis** — segments FP alerts by transaction type and amount bucket
4. **SLA Compliance Tracking** — case resolution vs. P1/P2/P3 SLA targets
5. **Chargeback Anomaly Detection** — accounts with abnormal chargeback ratios

**Scope note:** these queries model the *operational monitoring layer* of a live fraud system — a production-style schema with accounts, timestamps, priority tiers, chargebacks, and recovery flags, drawn from real fraud-investigation experience. This is deliberately **separate** from the Kaggle `creditcard.csv` used by the Python model, which is fully anonymised (V1–V28 PCA components, Amount, Time, Class) and has no accounts or timestamps. The Python side models *detection*; the SQL models the *monitoring* that would wrap it in production.
